🔹 1. 提高并发能力 • 阻塞查询：在传统的 JPA 或 JDBC 中，执行数据库查询时会 阻塞当前线程，直到查询完成并返回结果。对于每个请求来说，服务器必须为其分配一个线程，而在高并发环境下，线程池很容易被耗尽，导致请求排队，增加延迟。 • 非阻塞查询：在 WebFlux 中，当你查询数据库时，查询操作会返回一个 Mono 或 Flux，而这个操作会在 后台异步执行，不会占用当前线程。这样，当前线程可以去处理其他请求，极大地提升了 服务器的并发能力，因为每个请求都不再占用一个线程，线程池的压力得到了缓解。 🔹 2. 更高效的资源利用 在传统阻塞模型中，每个请求都需要 独占一个线程，即使线程在等待数据库查询或外部服务响应时处于空闲状态，也会占用计算资源，降低整体资源利用率。 • WebFlux 非阻塞模式：线程池中的线程会被 复用 来处理其他请求，特别是当请求正在等待 I/O 操作（如数据库查询或外部 API 响应）时，线程并不被阻塞，而是可以自由地处理其他的请求。 ⸻ 🔹 3. 更低的响应延迟 在传统的阻塞模式中，如果请求涉及多个 I/O 操作（如多次数据库查询、调用外部服务等），每个操作都会 串行执行，导致 总的响应时间增加，特别是在高负载情况下，延迟会显著提高。 • WebFlux 非阻塞模式：不同的 I/O 操作可以 并行执行，并且它们都不会阻塞当前线程，因此多个数据库查询或外部服务调用可以 同时进行，响应时间大大缩短。 4. 更好的可扩展性 • 传统的阻塞查询：在负载增加时，你可能需要 扩展服务器的硬件资源（如增加更多的线程池），但是即使增加硬件资源，阻塞线程的 扩展性有限，并且容易遇到性能瓶颈，特别是 I/O 密集型应用。 • WebFlux 的非阻塞查询：因为 线程不再被阻塞，在处理大量并发请求时，WebFlux 可以通过 更少的线程处理更多的请求，即使在 高并发场景下也能保持较低的延迟，系统的 可扩展性和处理能力大大增强。
